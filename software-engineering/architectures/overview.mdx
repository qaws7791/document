## 소프트웨어 아키텍처란

아키텍처: 어떠한 시스템의 형태

- 개발 관점 - 여러 명이 함께 개발할 때 어려움이 없어야 한다
- 배포 - 쉽게 배포할 수 있어야 한다
- 운영 - 비용 효율적이어야 한다. 운영 방식을 드러낼 수 있다
- 유지보수 - 빠르고 안전하게 기능이나 결함을 수정할 수 있다

- **중요하지 않은 선택사항**에 대해서는 가능한 많은 선택지를 열어두는 방향으로 전략을 취해야 한다
- 정책 - 시스템에서 **정책**을 가장 핵심적인 요소로 식별
- 세부 사항 - 세부 사항(구현)은 정책과 무관하게 만들 수 있는 형태의 시스템 구축
- 정책을 먼저 다 만들고 가장 마지막에 정책에 적합한 세부 사항을 선택

## 2. 유니터리 아키텍처

- 단일 머신에 초점, 초기 PC시대, 임베디드 시스템

유니터리 아키텍처란 모든 기능이 단일 코드베이스와 실행 환경 내에서 구현되는 아키텍처 스타일을 말합니다. 이 접근 방식은 코드의 일관성을 유지하고, 단일 배포 단위로 관리하기 용이하지만, 시스템이 커지면 유지보수와 확장성에 어려움을 겪을 수 있습니다.

## 분산형 아키텍처의 어려움

1. 네트워크는 믿을 수 없다 - 서비스 간 통신 시 네트워크 문제로 인해 장애가 발생할 수 있다
2. 레이턴시가 존재한다 - 모놀리식에 비해 서비스 간 통신 시 레이턴시가 길다
3. 대역폭은 무한하지 않다 - 서비스 간 통신이 많아질 경우 대역폭으로 인해 문제가 발생할 수 있다
4. 네트워크는 안전하지 않다 - 네트워크를 통해 공격당할 가능성이 더 많아진다
5. 네트워크 토폴로지는 항상 바뀐다 - 네트워크에 대한 변경 사항을 지속적으로 인지해야 한다
6. 네트워크 관리자는 여러 명이다 - 네트워크를 다루기 위해 많은 협업이 필요하다
7. 원격 액세스는 돈이 든다 - 모놀리식에 비해 네트워크 및 하드웨어 자원이 더 많이 소모된다
8. 네트워크는 균일하지 않다 - 표준이 존재하지만, 다양한 하드웨어들이 모든 상황에서 동일하게 잘 작동할 지는 알 수 없다
